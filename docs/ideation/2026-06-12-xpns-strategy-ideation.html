1.
Trust Trifecta: P1 Pre-Launch Bundle
Submission experience Must-ship · pre-demo gate 95% conf Cx Low
Description
Three open P1 bugs are not independent defects — together they constitute the tamper-evident receipt chain that makes xpns's core claim credible: "receipt captured at moment of expense, immutable, with full audit trail." Fix all three before the demo.

P1-B (storage rules): Add uid segment to receipt path (receipts/{orgId}/{uid}/{expenseId}) and tighten the write rule to request.auth.uid == uid. Currently any org member can overwrite any receipt — which voids the audit trail claim entirely.

P1-C (orphan race): Invert the write order: create the Firestore expense document first (with status: uploading), upload to Storage second, then update the doc with the storage path. If Storage fails, the doc can be cleaned up or retried. No phantom receipts.

P1-E (client-controlled orgId): The invite Cloud Function must mint a signed token with orgId embedded server-side; the join function validates the token signature. orgId must never come from the client payload. Closes the org membership injection attack surface.
Basis
directP1-B, P1-C, P1-E explicitly documented in to-fix.md (code review 2026-06-11). Each has a one-day fix. Together they constitute the product's trust foundation.
Rationale
xpns sells on "receipt at moment of expense, tamper-evident." None of that claim holds while P1-B (peer can overwrite receipts), P1-C (orphan phantom records), and P1-E (org injection via client payload) are open. These are not cosmetic bugs — they are the attack surface on the product's core proposition. Showing the dashboard to a demo prospect while any of these are open is a liability.
Downsides
Fixing P1-E requires a Cloud Function rewrite of the invite flow — medium scope for a 9-day window. P1-B requires a storage path migration if any receipts already exist in prod. P1-C requires changing the write order in the submission flow (test coverage needed).
2.
WhatsApp-First Full Lifecycle
Submission experience Admin visibility Reimbursement flow 90% conf Cx Medium
Description
The entire xpns loop lives in WhatsApp. The PWA becomes the power-user escape hatch, not the primary surface. Four touchpoints, all in WhatsApp:

Submit: Employee sends photo + caption ("taxi R$45 client meeting") to a dedicated WhatsApp Business number. A Cloud Function webhook parses it, creates the expense doc, attaches the image, and replies with a confirmation: "Got it. R$45 · transport · pending review."

Approve: Admin receives a WhatsApp message: amount, category, employee name, receipt thumbnail link. Reply "ok" approves; reply "no [reason]" rejects with a note back to the employee. Dashboard reflects it instantly.

Pay: After approval, admin initiates reimbursement via the employee's pre-registered Pix/SPEI/Nequi handle (see idea #5). xpns sends a payment confirmation to both parties.

Status push: Employee receives a WhatsApp notification at each status transition: Submitted → Approved → Paid. No need to open the app to check status.
Basis
externalReme.ai: WhatsApp approval integration raised completion rate from 73% to 96% and cut time from 3-5 days to under 4 hours. EnKash/ZYNO Expenz (India, structurally identical market) ships WhatsApp-first expense submission. Fogg Behavior Model: each extra step −20% completion; removing "open app" from both employee and admin paths is the highest-leverage single change. WhatsApp penetration: 90% daily in Brazil.
Rationale
The status quo is WhatsApp + spreadsheets. xpns is the structured layer on top of the same channel employees and admins already live in. Any design that requires either party to leave WhatsApp creates a behavior-change friction that kills adoption before habits form (Fogg: 18-254 days). The WhatsApp Business API is the channel; the PWA holds reports, exports, and settings. This is the single change most likely to move both submission rate and approval rate — the two key metrics in STRATEGY.md — simultaneously.
Downsides
WhatsApp Business API requires a verified business account and per-message pricing (Meta's template costs, ~$0.005-0.02 per message depending on country). Cloud Function webhook integration needed. Free-tier conversation window (24h reply window for user-initiated) needs to be managed carefully for async approval flows. Not available in demo without the API setup.
3.
Submission Health Score + Running Balance
Admin visibility 88% conf Cx Low
Description
Two dashboard-first changes that together replace the admin's spreadsheet as the operating view — and directly track STRATEGY.md's key metrics.

Submission Health Score: One prominent KPI at the top of the admin dashboard: (expenses with receipt attached / total expenses logged) × (approved first-pass / submitted this period). Green/yellow/red thresholds. No ML, no new data — pure Firestore aggregate math on the data that already exists. Admin logs in on Monday and sees in one glance whether the team's receipt habits are healthy.

Running Balance: Remove the concept of a periodic "expense report." Instead, the dashboard shows a live per-employee owed balance. Admin sees "Carlos: R$340 owed, 3 approved." Expense closes when marked paid. No month-end compile step — the balance is always current.
Basis
directSTRATEGY.md defines submission rate and approval rate as the two key product metrics. Making those metrics visible in the product creates the behavioral feedback loop that improves them. The date filter (P1-A) must be fixed as a prerequisite — composite Firestore index + where() wiring — before any aggregate is credible.
Rationale
Admins who can see their own health score log in weekly to manage it. Weekly login frequency is the strongest predictor of 90-day org retention. The score also becomes a demo proof point and, later, a benchmark between orgs. The running balance removes the "report" concept entirely — the report is a periodic-batch artifact from the spreadsheet world. If data is in Firestore in real time, there is no report; there is a current state.
Downsides
P1-A (date filter no-op) must ship first — the health score is meaningless if the underlying aggregate cannot be computed correctly. Requires composite Firestore index creation on (orgId, status, createdAt). Score definition needs team agreement on what "healthy" thresholds mean for a new org vs. a 6-month-old one.
4.
Threshold Auto-Approve + Admin Triage Board
Admin visibility 85% conf Cx Medium
Description
Invert the approval default: admin opts in to review, not out of it. Per-org configurable threshold (e.g. R$150 / MXN 500). Expenses below it auto-approve after 24 hours unless the admin explicitly rejects. Admin is only pulled in for anomalies.

The admin dashboard becomes a triage board — three queues:
  • Green — below threshold, receipt attached, auto-approving in Nh. No action required.
  • Yellow — above threshold or flagged category. Needs review.
  • Red — missing receipt, over-limit, or rejected. Action required.

Structural analog: the pharmacist refill authorization model — routine refills under a defined limit dispense automatically with a physician veto window. Only exceptions require active approval.
Basis
reasonedB2B tools: 25% avg adoption vs 97% for embedded-workflow tools. Admin at a LATAM SMB with no finance team is already managing operations, sales, and logistics — adding "review every expense" as a daily task is the adoption killer. Reducing required admin actions to exceptions preserves control while removing friction. externalPharmacist refill authorization model (managed care): routine cases dispense automatically; physician veto window preserves oversight. Direct structural parallel.
Rationale
Approval rate (% approved first review without back-and-forth) is a key metric. Auto-approve makes routine approval invisible — it happens without admin action. Admin trust is maintained by the veto window and the triage board's transparent exception queue. Reducing admin load is the primary lever on org retention at 30/90 days.
Downsides
Trust barrier: admins may not trust auto-approval until they've used the product long enough to calibrate the threshold. Needs clear UI indication of what's auto-approving and when. Potential abuse if employees learn the threshold (split one large expense into two small ones). Fraud surface is real at scale but minimal for 5-50 person orgs.
5.
Local Rail Reimbursement Closure
Reimbursement flow 82% conf Cx High
Description
Close the reimbursement loop inside xpns using the dominant free instant payment rail of each LATAM country. Admin pre-registers each employee's payment handle once. After approval, admin taps "pay" and xpns initiates the transfer (or deep-links into their banking app pre-filled). Employee receives a WhatsApp confirmation when paid. Expense closes.

Country rail targets: Brazil → Pix (CPF or phone key, free, instant, 63B transactions in 2025). Mexico → SPEI (CLABE account number, near-instant). Colombia → Nequi (phone number, instant). Chile → WebPay / CuentaRUT.
Basis
externalMarket gap confirmed by web research: no tool combines LATAM-local + reimbursement-first + WhatsApp + local payment rails. Pix recorded 63 billion transactions in 2025 (free, instant); projected to overtake credit cards by 2027. Reme.ai data: cutting reimbursement cycle from days to hours is the #1 stated differentiator. Behavioral research: employees who get reimbursed fast submit more — the payment closes the habit loop.
Rationale
"Will I get my money back?" is the emotional core of expense submission. Slow reimbursement is the primary reason employees stop submitting (why bother if it takes two weeks and requires chasing?). Pix removes the friction entirely: approval-to-paid in under 10 minutes. This is the move that makes xpns genuinely better than the spreadsheet — not just more organized, but faster money in pocket.
Downsides
High complexity: requires per-country API integrations or a payments middleware (Belvo, Fintoc, Kushki). Regulatory compliance differs by country (KYC requirements for initiating bank transfers). Not MVP-feasible for 2026-06-30 launch without a partnership or existing payment provider. Start with Brazil/Pix as the lowest-friction path; deep-link approach (pre-filled bank transfer in native banking app) is lower-risk than direct API initiation.
6.
In-Context Spend Nudge
Submission experience 72% conf Cx Low
Description
When an employee mentions a spend in WhatsApp — keyword pattern ("compré", "pagué", "restaurant", "taxi", "gas", or an amount pattern "R$", "MXN", "$") — xpns sends a 1-tap "snap it now" prompt within seconds. Receipt captured in context, not reconstructed from memory days later. Also available as a manual trigger: employee texts the xpns bot channel with an amount and immediately gets the photo request.

Structural analog: field logistics dead-drop checkpoints — long-haul trucking requires GPS waypoint pings at fixed intervals because the compliance window closes fast. The mechanic transfers: prompt at the moment of the event, not at end-of-day batch.
Basis
externalField logistics checkpoint pattern (DHL convoy tracking): low-tech workers + compliance requirement + moment-of-action capture window. Structural parallel: expense submission abandonment happens because the physical receipt is no longer accessible hours or days later, not because the employee is unwilling to submit. reasonedFogg Behavior Model: motivation is highest immediately post-purchase (the event just happened, the receipt is in hand). Each hour that passes drops both motivation and ability (receipt lost, amount forgotten).
Rationale
Submission rate is a key metric. The hardest submission problem is not friction in the app — it's that employees delay until end-of-week and then can't reconstruct the receipts. An in-context nudge converts the spend moment into the submission moment. Depends on WhatsApp Business API (see idea #2) — this is a feature of the WhatsApp channel, not the PWA.
Downsides
Keyword detection in Spanish/Portuguese is noisy — many messages containing "taxi" are not expense events. False positives annoy employees and kill trust in the bot. Conservative approach: only trigger on opt-in keyword pattern from the employee's own messages in the org group, or on explicit employee-initiated messages to the xpns bot number. Depends on WhatsApp Business API being in place (idea #2 prerequisite).
7.
Bottom-Up PLG: Employee Receipt Triggers Org
Org growth 68% conf Cx Medium
Description
Invert the acquisition funnel: org creation is triggered by employee action, not admin setup. Two entry points:

Employee receipt as org seed: Employee submits a receipt (WhatsApp or web). System detects no linked org. Creates a draft org and sends the admin an activation link: "Carlos from [company] just submitted a R$45 expense and is waiting for your approval. Click to claim your org and reimburse him." Admin's first interaction is already valuable — there's money on the table, not a signup form.

Invite link as viral unit: When admin invites a team member, the invite lands on a co-branded page ("Maria from Acme is using xpns — join to get reimbursed faster"). Employees who join under one org become a referral seed: if they later start their own company, account is pre-created. Org creation pre-fills from their profile.
Basis
reasonedPLG mechanics: removing the "admin configures first" gate collapses the setup funnel entirely. Admin's first interaction is already valuable (pending reimbursement), not a signup form. Converts the pain point directly into a signup trigger. externalLoom's viral loop: each video sent creates a passive referral (recipient must create account to reply). Structural parallel: each expense submitted creates an admin activation trigger.
Rationale
Org growth at 90/365 days depends on acquiring paying orgs. With no sales team at launch, every acquisition mechanism matters. Bottom-up PLG turns employees into passive salespeople — they submit, which means admins get notified, which means admins activate. This is the structural move that makes each employee invite a potential org acquisition rather than just a user onboard.
Downsides
Employee may submit to a wrong or non-existent org. Draft org creation could be spammy if not guarded (e.g., employee must confirm the company name/admin email before the activation link fires). The viral unit (invite page) requires design and copy investment to convert — a generic invite URL does not carry the right emotional hook.
